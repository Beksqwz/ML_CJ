ML_CJ - project cleanup and pipeline notes
==========================================

Date: 2026-07-11

Purpose
-------
This file documents what was changed in the ML_CJ repository, what data was
prepared, what sources were used, and which files should be used next.


Repository cleanup
------------------
I cleaned the repository so it is easier to upload and review:

0. Flattened the project layout.
   The previous nested ML_CJ contents were moved to the project root, so scripts,
   data, reports, raw files, road files, and documentation now live at one level.
   The outer Windows folder was requested to be renamed from "RoadRisk AI" to
   "ML_CJ", but Windows reported that the folder is in use by another process.
   Close IDE/terminals that point to the folder, then rename it manually or run:
   Rename-Item -LiteralPath "C:\Users\yedil\OneDrive\Рабочий стол\RoadRisk AI" -NewName "ML_CJ"

1. Removed generated Python cache files:
   - scripts/__pycache__/
   - *.pyc

2. Removed duplicate CSV exports for generated datasets.
   Parquet is now the canonical data format for processed/external data.
   CSV files can be regenerated by the scripts when needed.

3. Removed old duplicate report folders from earlier repeated runs.
   Kept the latest useful reports for each important stage.

4. Added .gitignore:
   - ignores Python caches;
   - ignores local env/dependency folders;
   - ignores generated CSV exports under data/processed and data/external.

5. Added requirements.txt with main Python dependencies.

6. Replaced the empty scripts/historical_accidents.py with a small source-file
   sanity check. It verifies that astana_accidents.csv and
   astana_accidents.parquet exist.


Important note about CSV files
------------------------------
The repo still keeps astana_accidents.csv because it is one of the raw source
artifacts from stage 1. Generated processed CSV files were removed because the
matching Parquet files are smaller and safer for ML pipelines.

Stage 3 - Accident data cleaning
Status: done.

Main script:
  - scripts/cleaning_accidents.py

Main output:
  - data/processed/astana_accidents_clean.parquet

What it does:
  - parses rta_date;
  - creates accident_date and accident_datetime;
  - converts EPSG:3857 coordinates to longitude/latitude;
  - creates time features;
  - removes unusable records and duplicate objectid values.


Stage 4 - Road matching
Status: done.

Main scripts:
  - scripts/match_accidents_to_roads.py
  - scripts/create_ml_base_accidents.py
  - scripts/filter_ml_base_accidents.py
  - scripts/normalize_road_features.py

Main outputs:
  - data/processed/accidents_with_roads.parquet
  - data/processed/accidents_with_roads_ml_base.parquet
  - data/processed/accidents_with_roads_ml_filtered.parquet
  - data/processed/accidents_with_roads_ml_suspicious.parquet
  - data/processed/accidents_with_roads_ml_ready.parquet

Important final file for the next ML stages:
  - data/processed/accidents_with_roads_ml_ready.parquet

Final checked numbers:
  - raw accidents: 13,663 rows
  - cleaned accidents: 13,663 rows
  - matched accidents: 13,663 rows
  - ML-ready accidents after distance <= 300 m filter: 12,449 rows
  - road_segment_id missing in ML-ready: 0
  - objectid duplicates in ML-ready: 0
  - leakage columns in ML-ready: 0
  - road_lanes_num missing: 0
  - road_maxspeed_kmh missing: 0
  - validation status for ML-ready: READY

Road matching policy:
  - high confidence: distance_to_road_m <= 100
  - medium confidence: 100 < distance_to_road_m <= 300
  - low/suspicious: distance_to_road_m > 300

The ML-ready file keeps high and medium confidence rows.

  - weather hourly rows: 127,176

  - weather missing values: 0
  - OSM POI rows: 5,883

External sources used:
  - Open-Meteo Historical Weather API
    https://open-meteo.com/en/docs/historical-weather-api

  - OpenStreetMap via Overpass API
    https://overpass-api.de/

  - Python holidays package for Kazakhstan public holidays
    country_holidays("KZ", observed=True)

Stage 5 limitations:
  - weather is city-level hourly ERA5 data for Astana, not per-road microclimate;
  - visibility is not included in the selected Open-Meteo variables;
  - school calendar features are heuristic, not official year-specific dates;
  - POI data comes from current OpenStreetMap and may be incomplete.


Recommended next stage
----------------------
Next roadmap step: stages 6-7.

Build a dataset where:
  - one road_segment_id + one hour = one row;
  - target = 1 if an accident happened on that segment/hour;
  - target = 0 otherwise.

Use this as the main accident input:
  - data/processed/accidents_with_roads_ml_ready.parquet

Join these external sources later:
  - data/external/calendar_features_hourly.parquet by datetime_hour;
  - data/external/weather_astana_hourly.parquet by datetime_hour;
  - data/external/pois_astana_osm.parquet after aggregating POIs to road segments.


Suggested command order
-----------------------
From the project root:

1. Check raw source files:
   py scripts/historical_accidents.py

2. Clean accidents:
   py scripts/cleaning_accidents.py

3. Validate clean accidents:
   py scripts/validate_historical_accidents.py --input data/processed/astana_accidents_clean.parquet

4. Match accidents to roads:
   py scripts/match_accidents_to_roads.py

5. Create leakage-free ML base:
   py scripts/create_ml_base_accidents.py

6. Filter road match quality:
   py scripts/filter_ml_base_accidents.py --max-distance-m 300

7. Normalize road features:
   py scripts/normalize_road_features.py

8. Create calendar data:
   py scripts/create_calendar_features.py

9. Fetch weather:
   py scripts/fetch_open_meteo_weather.py

10. Fetch OSM POIs:
   py scripts/fetch_osm_pois.py

11. Validate external data:
   py scripts/validate_external_data.py
